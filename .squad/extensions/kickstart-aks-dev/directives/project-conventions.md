# Project Conventions

Kickstart-specific rules that override or extend the squad defaults. Every agent must read this before starting work.

> **North star:** the v2 design lives in [`docs/v2-implementation-brief.md`](../../../../docs/v2-implementation-brief.md). All new code, agents, and tooling must align with the harness + packs architecture described there. If a brief section and an older convention disagree, the brief wins. Flag the conflict in a decision note.

---

## Architecture invariants (non-negotiable)

- **Two-layer model:** `@kickstart/harness` is runtime glue only. All domain knowledge lives in packs (`pack-core`, `pack-azure`, `pack-aks-automatic`, `pack-github`). Never add domain logic to the harness.
- **Primitives only:** Pack, Agent, Skill, Tool, UserAction, Component, Guardrail. No new primitive types without a decision note.
- **Authoring formats:** agents use `.agent.md` (VS Code custom agent format). Skills use `SKILL.md` (agentskills.io). Do not invent new formats.
- **A2UI streaming:** UI renders only through the `core.emit_ui` tool call. No envelopes. No JSON parsers on the client.
- **No v1 concepts in new code:** no phases, no FSM, no stepwise gates, no `setup-generation.ts`, no integration-kit prompts. Deleting legacy paths is allowed when they block v2 work (link the PR to the brief section).
- **In-tree only:** third-party pack loading is out of scope for v2. All packs live under `packages/` in this repo.
- **Sealed registry:** packs register at startup, then the registry is sealed. Runtime mutation is a bug.

## Branching & PRs

- **Branch naming:** `squad/{issue-number}-{kebab-case-slug}` (e.g., `squad/42-pack-azure-skill-resources`). Squad-internal work without an issue uses `squad/<role>-<slug>`.
- No direct pushes to `main`. All work goes through PRs.
- Always open PRs as **draft** first. Mark ready only after CI is green.
- Always **rebase** onto `main`. Never create merge commits on the PR branch.
- **Squash-merge** and delete the branch on merge.

## Documentation (mandatory, not optional)

- Every user-facing change updates [`docs-site/docs/`](../../../../docs-site/docs/) in the same PR. No "docs will follow" PRs.
- Every architectural change updates or cites a section of `docs/v2-implementation-brief.md`. If the brief is wrong, update the brief in the same PR.
- Public API changes (HTTP endpoints, SSE event shapes, tool schemas) update [`docs/api-reference.md`](../../../../docs/api-reference.md).
- Pack additions get a short page in `docs-site/docs/extending/` describing the pack's agents, skills, tools, user actions, and components.
- See the `docs-changelog` skill for the exact checklist.

## Changelogs & releases

- Every PR that ships user-observable behaviour includes a **changeset** (`npm run changeset`). CI surfaces missing changesets via `changeset status`.
- Changesets describe user impact in one line. Not the implementation detail.
- `CHANGELOG.md` files are generated by `@changesets/cli`. Do not hand-edit them.
- All packages version in lockstep (linked in `.changeset/config.json`).
- The Static Web App (SWA) is the **pre-prod** environment. Every push to `main` deploys it.
- Tag pushes (`v*`) mark versioned releases and cut GitHub Release notes. They do not trigger a separate production deploy today.
- Release PRs are owned by the Lead (Leela) with Scribe curating the release notes.
- See the `release-process` skill for the exact release workflow.

## Work tracking

- GitHub Issues is the source of truth.
- Every issue gets a milestone (or explicit `Backlog`) during triage.
- Project board stages: Backlog → Ready → In progress → In review → Done.
- Assign issues to @sabbour. Agents cannot be assignees.
- Post major findings as issue comments as work progresses. Not just in decisions.md.

## Testing

- Vitest unit tests for packs and harness.
- Playwright E2E for user-facing flows under `packages/web/e2e/`.
- Pack conformance tests are required for every new pack (see `testing-strategy` skill).
- All CI checks must pass before merging. Do not merge PRs with failing required checks.

## Tooling & models

- Claude for code-writing work (implementation, refactoring, bug fixes, test code).
- GPT-5 class model for security review (Zapp) and for stuck architectural calls.
- `@openai/agents` SDK is the runtime for v2. Do not add a second agent framework.
- Changesets is the only supported versioning tool.

## UI standards

- **Fluent UI React v9 only.** No custom CSS class systems, no inline styles.
- Style with `makeStyles()` + design tokens. Theme through the `resolvedTheme` pattern.
- No emoji in shipped UI. Use Fluent icons or Material Symbols.
- Components registered in the A2UI catalog must be deterministic and side-effect-free at render time. Data fetching goes through SDK tools, not React effects that hit networks.

## Security

- OIDC credentials use `secrets.*`, never `vars.*`, in GitHub Actions.
- No inline `<script>` in `index.html`. CSP `script-src 'self'` stays clean.
- No CDN script tags. Everything goes through the bundler.
- Pack guardrails are declared in the pack, not patched at call sites.
- Tool schemas are the security surface. Never widen a tool's accepted args without a Zapp review.
- Zapp reviews every PR touching: authentication, CORS, secrets, ARM proxy, GitHub OAuth, pack guardrails, new SDK tools.

## Retrospectives

- Retros include wall-clock time vs estimates per issue.
- Capture review-loop inefficiencies and propose concrete fixes (not vibes).
- Velocity metrics flow into the next sprint's estimate calibration.
